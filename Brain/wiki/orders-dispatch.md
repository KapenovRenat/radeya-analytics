# Фича: Заказы — таблица, детали, авто-отправка поставщику

> Раздел Операционка → Заказы: таблица заказов после графиков, модалка «Подробно», и авто-отправка нового заказа поставщику в Telegram (по матчингу артикул→товар→поставщик), с настройками таймера/cron в админке.

## Содержание

### Где
`/stores/[id]/orders` — таблица **после** существующих графиков (через `<Section>`). Текущий API `/orders` отдаёт только агрегаты для графиков — остаётся, добавляются новые роуты.

---

### Фаза 1 — Таблица + «Обновить 14 дней» + модалка «Подробно» ✅ ГОТОВО

> Реализовано 2026-06-23. Без миграций (существующие таблицы). Кнопка «Обновить 14 дней» синкает заказы → потом состав (entries). Те же 2 фазы добавлены к кнопкам на `/sync`.
> Файлы: `lib/kaspi/order-status.ts`, `app/api/.../orders-list/route.ts`, `app/api/.../orders/[orderId]/route.ts`, `app/stores/[id]/orders/orders-table.tsx`.

**API** `GET /api/kaspi/stores/[id]/orders-list?page=&search=&status=` — пагинация 20/стр, поиск (номер/клиент), фильтр статуса.

**Колонки:** № заказа · Дата поступления · Сумма · Статус · Состояние · Клиент · Город · «Подробно» · «→ Поставщику» (Фаза 2).

**Маппинг статусов** (Kaspi `status`+`state`+`waybillNumber` → отображение):
- **Новый** — нет накладной (`waybillNumber` пустой); `NEW/APPROVED_BY_BANK`
- **Принят** — накладная появилась (`waybillNumber` есть) / `ACCEPTED_BY_MERCHANT`
- **На доставке** — доехал до ПВЗ (`state=KASPI_DELIVERY/DELIVERY`, статусы доставки)
- **Завершён** `COMPLETED` · **Отмена** `CANCELLING` · **Отменён** `CANCELLED` · **Возврат** `RETURNED`

**Состояние** = `state`: `KASPI_DELIVERY`→«Kaspi Доставка», `ARCHIVE`→«Архив».

**Кнопка «Обновить за 14 дней»** над таблицей — синк последних 14 дней (обновить статусы + новые). Год синкается отдельно (страница `/sync`). Переиспользует `useSync(days=14)`.

**Модалка «Подробно»** (`GET /api/kaspi/stores/[id]/orders/[orderId]` → заказ + `kaspi_order_entries`):
блоки Состав (товар/код/кол-во/цена) · Заказ (даты, оплата, предзаказ, собран) · Доставка (способ, Kaspi доставка, стоимость, ПВЗ, город, адрес, передача курьеру) · Клиент (имя, телефон).

### Фаза 2 — Матчинг + сообщение + ручная отправка + статус

**Матчинг:** позиция заказа `offerCode` (артикул) → `products.code` в нашей базе ТОВАРЫ → `products.supplier` → контакт в `suppliers` (группа/личка). Группируем позиции по поставщику.

**Куда слать:** приоритет `tgGroupId` → группа, иначе `tgChatId`. В группу бот может слать без `/start` (нужно только добавить бота).

**Сообщение** — на каждую позицию отдельно, через Telegram **`sendPhoto`** (caption = текст, photo = `products.imageUrl` из Cloudinary; картинка **обязательна**):
```
ЗАКАЗ # {номер}
🚨 Каспи магазин 🚨
Отгрузка {город склада = originAddressCity}
Дата сдачи: {передача курьеру} ✅

{Название — распарсенное}
Основная ткань: {из квадратных скобок [...]}
Артикул изделия: {код}
Доп: {редактируемый текст из настроек}
```

**Парсинг `products.name`** (эвристика, тюним на реальных данных):
- Полное: `RADEYA Буйвол, 33x59x42 см, коричневый [ткань Milos16/Milos 02]`
- **Название** = до «см» включительно → `RADEYA Буйвол, 33x59x42 см` (цвет отбрасываем)
- **Ткань** = содержимое `[ ]` → `ткань Milos16/Milos 02`

**Таблица `order_dispatches`:** `orderId, orderCode, supplierName, target(group/chat), status(sending/sent/failed), error, sentAt`. Уникальный индекс `(orderId, supplierName)`. Одна строка на (заказ × поставщик). Заказ «отправлен» → кнопка в таблице скрывается, показываем «✓ Отправлено».

**Анти-дубль (защита от повторной отправки, в т.ч. гонки cron):**
- Перед отправкой — идемпотентная вставка `INSERT ... ON CONFLICT (order_id, supplier) DO NOTHING RETURNING id`.
- Вернулся id → мы захватили заказ, шлём, ставим `sent`. Конфликт → уже отправлен/отправляется → пропускаем.
- Cron отбирает только заказы **без** строки в `order_dispatches`. Ручная кнопка пишет туда же → cron потом не трогает. → заказ уходит **ровно один раз**.

**Ручная кнопка** «→ Поставщику»: `POST /orders/[orderId]/dispatch` → матчинг+отправка сейчас.

### Фаза 3 — Cron авто-отправка

**Эндпоинт** `POST /api/cron/dispatch` (защита `CRON_SECRET` в заголовке). За вызов:
1. Синк заказов за **14 дней** (orders + entries), обновить статусы/новые
2. На каждый новый заказ — таймер из настроек (по умолч. 1 ч от `creation_date`)
3. Прошёл таймер + авто-отправка вкл → отправить поставщику → `order_dispatches`, больше не слать

**Crontab на VPS** — дёргает эндпоинт **каждую 1 минуту**:
```
* * * * * curl -s -H "x-cron-secret: XXX" https://analytics.radeya.kz/api/cron/dispatch
```
Эндпоинт сам gate-ит по `cronIntervalMin` (работает только если прошло ≥ N мин с `lastCronRunAt`) — так интервал меняется из админки без правки crontab.

### 📌 РЕШЕНО — «Дата сдачи» в сообщении = передача курьеру − 1 день

**Решение (2026-06-26):** `Дата сдачи = courierTransmissionPlanningDate − 1 день` (Asia/Almaty).

Почему так: «Планируемой даты прибытия» (которую хотели изначально, `прибытие − 1`) **нет** в Kaspi orders API (`/orders`) — проверено по сырому `kaspiDelivery`, там только `courierTransmissionPlanningDate` (передача курьеру) и `plannedDeliveryDate` (доставка). Формулой прибытие не вывести (разрывы дат у заказов разные). Поэтому берём надёжное поле из API — **передачу курьеру** — и вычитаем 1 день.

Пример (заказ 5732): передача курьеру `2 июл 20:00` − 1 = **1 июл** (совпало с прибытием 1 июл ✅).

⚠️ **Ещё не реализовано в коде.** Сейчас `lib/dispatch/send-order.ts` считает `Дата сдачи = plannedDeliveryDate − дни склада` — **это надо заменить** на `courierTransmissionPlanningDate − 1 день`. Дни склада из Excel для этой даты больше не используем. Ждём «делай».
