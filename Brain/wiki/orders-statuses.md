# План: статусная модель заказов (как в Kaspi/МойСклад) + маршрутизация поставщикам

> Цель — привести наши статусы заказов к виду, как в МойСкладе Рената (он связан с Kaspi), чтобы корректно решать, **кому и когда отдавать заказ** (реальный поставщик / внутренний склад / своя доставка).

## Содержание

### A. Что реально приходит от Kaspi (поля, которые у нас есть)

В `kaspi_orders` (колонки) + `raw_data` (jsonb):

| Поле | Где | Значения / смысл |
|---|---|---|
| `status` | колонка | `APPROVED_BY_BANK`, `ACCEPTED_BY_MERCHANT`, `COMPLETED`, `CANCELLING`, `CANCELLED`, `RETURNED` |
| `state` | колонка | `NEW`, `SIGN_REQUIRED`, `PICKUP`, `DELIVERY`, `KASPI_DELIVERY`, `ARCHIVE` |
| `deliveryMode` | колонка | `DELIVERY_PICKUP` (самовывоз), `DELIVERY_LOCAL`, `DELIVERY_REGIONAL_*` |
| `isKaspiDelivery` | колонка | true = Kaspi Доставка, false = своя доставка/самовывоз |
| `waybillNumber` | колонка | накладная появляется после принятия |
| `assembled` | колонка | товар собран/упакован |
| `preOrder` | raw_data.attributes | **true = предзаказ** (нет в наличии), false = из наличия |
| `kaspiDelivery.courierTransmissionDate` | raw_data | проставлен → передан курьеру |
| `kaspiDelivery.courierTransmissionPlanningDate` | raw_data | план передачи курьеру (для «даты сдачи») |
| `kaspiDelivery.returnedToWarehouse` | raw_data | вернулся на склад |
| `cancellationReason` | колонка | причина отмены |

⚠️ «Планируемой даты прибытия» в API **нет** (см. [[orders-dispatch]]).

---

### B. Два измерения статуса

Список в МойСкладе смешивает **две независимые вещи**:

1. **Тип доставки** (фиксируется при заказе, не меняется):
   - **Kaspi Доставка** — `isKaspiDelivery=true` (`state=KASPI_DELIVERY`)
   - **Самовывоз** — `deliveryMode=DELIVERY_PICKUP` (`state=PICKUP`)
   - **Своя доставка** — `isKaspiDelivery=false` и не самовывоз
2. **Стадия жизненного цикла** — Новый → (Предзаказ | Упаковка) → Передача → Переданы на доставку → Доставлен; ветки отмен/возвратов.

Плюс флаг **Предзаказ** (`preOrder`) — пересекается со стадией.

→ У нас в модели держим: **стадия** + **тип доставки** + **флаг preOrder**. Отображаемый статус = их комбинация.

### C. Полный список статусов (цель — как в МойСкладе)

| # | Статус | Цвет (МойСклад) | Условие из Kaspi |
|---|---|---|---|
| 1 | **Новый** | зелёный | `status=APPROVED_BY_BANK`, нет `waybillNumber`, `state=NEW` |
| 2 | **На подписании** | оранжевый | `state=SIGN_REQUIRED` |
| 3 | **Самовывоз** | серый | `deliveryMode=DELIVERY_PICKUP` / `state=PICKUP` (тип доставки) |
| 4 | **Своя доставка** | коричневый | `isKaspiDelivery=false` и не самовывоз (тип доставки) |
| 5 | **Предзаказ** | синий | принят (`ACCEPTED_BY_MERCHANT`/есть waybill) **и** `preOrder=true` |
| 6 | **Упаковка** | салатовый | принят **и** `preOrder=false` **и** `assembled=false` (в наличии, пакуется) |
| 7 | **Передача** | бирюзовый | `assembled=true` **и** `courierTransmissionDate=null` |
| 8 | **Переданы на доставку** | зелёный | `courierTransmissionDate` проставлен |
| 9 | **Ожидают возврата/отмены** | красный | `status=CANCELLING` |
| 10 | **Ожидают решения по возврату** | синий | возврат запрошен (⚠️ точный Kaspi-статус уточнить: `KASPI_DELIVERY_RETURN_REQUESTED`?) |
| 11 | **Возврат** | красный | `status=RETURNED` |
| 12 | **Доставлен** | зелёный | `status=COMPLETED` |
| 13 | **Отменён** | серый | `status=CANCELLED` |
| — | ~~Сценарии~~ | — | внутренняя автоматизация МойСклада, нам **не нужна** |

**Приоритет вычисления** (сверху вниз, первый сработавший):
`CANCELLED` → `RETURNED` → возврат-запрошен → `CANCELLING` → `COMPLETED` →
если принят: `preOrder?`→Предзаказ, иначе `courierTransmissionDate?`→Переданы, `assembled?`→Передача, иначе Упаковка →
иначе по `state`: `SIGN_REQUIRED`→На подписании, `PICKUP`→Самовывоз → иначе Новый.

Тип доставки (Самовывоз/Своя/Kaspi) держим **отдельным признаком**, т.к. он нужен для маршрутизации даже когда стадия = Упаковка.

---

### D. Матрица маршрутизации — кому отдавать заказ ✅ подтверждено 2026-06-26

**Логика в два вопроса:**
1. Товар надо изготовить (**предзаказ**) или он уже на складе (**наличие**)?
2. Если наличие — доставка **Kaspi** (через ПВЗ) или **своя по Астане**?

| # | Случай | Условие | Получатель в Telegram | Триггер-статус |
|---|---|---|---|---|
| 1 | **Предзаказ, любой город** | `preOrder=true` | **реальный поставщик товара** (по артикулу → `products.supplier`) — изготавливает, сам сдаёт на местный Zammler/ПВЗ | «Предзаказ» |
| 2 | **Наличие + Kaspi Доставка** | `preOrder=false`, `isKaspiDelivery=true` | **[Астана] Кладовщик** — упаковывает и везёт на Zammler/ПВЗ | «Упаковка» |
| 3 | **Наличие + Своя доставка** | `preOrder=false`, своя доставка | **Своя доставка (газелист)** — развозит по городу | «Упаковка» |
| 4 | **Отмена / Возврат** | `CANCELLING`/`CANCELLED`/`RETURNED` | уведомление получателю (если уже слали) | как сейчас `notifyCancellation` |

**Ключевое различие:** для внутренних получателей (Кладовщик, газелист) триггер — **НЕ предзаказ**, а статус **«Упаковка»** (наличие, принят) + тип доставки. Предзаказы у них не идут.

Связано: [[orders-dispatch]] — сейчас `DISPATCHABLE = {"preorder"}`, уходят только предзаказы реальным поставщикам. Внутренние получатели (Кладовщик/газелист) — новая ветка маршрутизации.

### E. Что менять в коде (когда возьмём в работу)

1. **`lib/kaspi/order-status.ts`** — расширить `DisplayStatusKey`: добавить `signing` (На подписании), `pickup` (Самовывоз), `own_delivery` (Своя доставка), `return_pending` (Ожидают решения по возврату). Поправить приоритеты по таблице C.
2. **Тип доставки** — отдельная функция `deliveryType(order)` → `kaspi | pickup | own` (из `isKaspiDelivery` + `deliveryMode`).
3. **Маршрутизация** — в `dispatchOrder`/cron: ветвление по таблице D (предзаказ→поставщик; наличие+город+тип→внутренний получатель).
4. **Внутренние получатели** — задача «свои поставщики» в [[orders-dispatch]].
5. **Фильтр статусов в UI** — список значений в `orders-table.tsx` привести к таблице C.
