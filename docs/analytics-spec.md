# Enterprise Analytics Spec — Kaspi Seller Cabinet

> Спецификация аналитического модуля radeya-analytics. Следует канонам
> Tufte, Few, Munzner и практике Vercel / Linear / Stripe dashboards.
> Документ — источник правды для имплементации 7 табов.

---

## 0. Принципы (каноны)

**Data-ink ratio.** Каждый пиксель должен нести информацию. Убираем grid lines низкого контраста, 3D-эффекты, градиенты без значения, легенды которые дублируются с рядом стоящими label-ами.

**Preattentive attributes.** Цветовой hue — для категорий (statuses, payment modes). Saturation/luminance — для непрерывных шкал (heatmap). Position и size — самые сильные, зарезервированы для главного сравнения.

**Bar chart Y-axis всегда от 0.** Никогда не обрезаем. Если разброс маленький — берём line chart или delta-индикатор.

**Time — всегда на X-оси.** Line / area для непрерывного времени, bar для дискретных периодов (недели, месяцы).

**Sort by value, not alphabet.** Все категориальные breakdowns сортируются по убыванию метрики. Единственное исключение — day-of-week (дни недели в порядке недели) и hour (0→23).

**No pie charts с ≥6 сегментами.** Донат OK для 2–4 категорий с явным whole. Для 5+ — horizontal bar chart.

**Единая цветовая палитра.** Один цвет = одна сущность во всех табах. Смена метрик не меняет цвет payment_mode: PREPAID всегда зелёный, PAY_WITH_CREDIT всегда фиолетовый.

**Reading gravity F-pattern.** KPIs слева-сверху. Главный график — выше фолда. Вторичные — ниже фолда. Детальные таблицы — последними.

**Accessible in 3 seconds.** На каждом табе должен быть KPI-заголовок, отвечающий на главный вопрос таба, читаемый за 3 секунды. «Выручка 44.7М ₸, +12% vs прошлый период».

## 1. Информационная архитектура

**8 табов (в порядке важности):**

## 2. Универсальные элементы

### 2.1 Глобальная панель фильтров (верх каждого таба, sticky)

1. **Селектор магазина** — только если магазинов >1. По умолчанию — «Все магазины».
2. **Date range preset chips** — 7д / 30д / 90д / 180д / 365д. По умолчанию **30 дней**.
3. **Кастомный date range picker** — для нестандартных периодов.
4. **Period granularity** — `daily / weekly / monthly`. Auto-detect по длине range:
   - ≤ 31 день → daily
   - 32–180 дней → weekly
   - > 180 дней → monthly
5. **Refresh / Sync** — кнопка с состоянием прогресса.

### 2.2 Универсальный chart-card

- **Title** (14px font-semibold) слева
- **Info icon tooltip** — объяснение метрики (не техническое — бизнес-смысл)
- **Right-side action** — export CSV, open full-screen (позже)
- **Body** — чарт, height 280–320px
- **Loading state** — skeleton с той же высотой (предотвращаем layout shift)
- **Empty state** — «Нет данных за выбранный период», с подсказкой «измените фильтр»

### 2.3 KPI-карточка (используется везде)

- Иконка + label + value + **delta-badge** vs предыдущий период
- Delta: ↑ зелёный (growth), ↓ красный (decline), → серый (stable ±2%)
- Hover на icon — tooltip с формулой метрики
- Tabular numbers для значений (одинаковая ширина цифр)

## 3. Таб 1 — Дашборд (overview)

**Задача таба.** Дать полную картину магазина одним экраном. Если клиент увидит только этот таб — он должен понять здоров ли бизнес.

### 3.1 Above the fold (P0)

**6 KPI-карточек с delta vs предыдущий период** (сравниваемый период = равный длительности предыдущий):

| Метрика | Формат | Источник | Почему важно |
|---|---|---|---|
| Выручка | 44.7М ₸ ↑12% | `SUM(total_price)` по COMPLETED | Главная цифра |
| Заказов | 1 342 ↑8% | `COUNT(*)` COMPLETED | Объём спроса |
| Средний чек | 29 006 ₸ ↑4% | `AVG(total_price)` | Качество продукта |
| % отмен | 8.6% ↓0.3пп | `CANCELLED / total` | Health signal — ≥3% = тревога |
| % возвратов | 3.9% → | `RETURNED / total` | Health signal — ≥2% = тревога |
| Уникальных клиентов | 820 ↑15% | `COUNT(DISTINCT customer_name)` | Охват |

**Визуальный hierarchy:** 2 ряда × 3 карточки на desktop (6 KPI). На mobile — 2 × 3 или 3 × 2.

### 3.2 Первый fold (P1)

**Площадной график выручки по дням + заказы вторым line.** Dual-axis area chart:
- X: date
- Y1 (left): выручка, градиентный area #6c5ce7 → transparent
- Y2 (right): заказов, тонкая line #00d2a0
- Tooltip при hover: обе метрики + delta vs предыдущий день

**Почему area, а не bar.** Пользователь смотрит тренд, не конкретные дни. Area передаёт «поток выручки» лучше, чем дискретные bars.

### 3.3 Ниже фолда (P2)

**Таблица «Топ-10 дней по выручке»** — 4 колонки:
- Дата (с днём недели!)
- Заказов
- Выручка
- Средний чек

Сортировка по выручке. Строки с day_of_week = Sunday/Saturday подсвечиваются тонким background — «в выходные бывают пики?».

**3 мини-карточки** в ряд:
- Лучший день недели (по средней выручке)
- Лучший час (по заказам)
- Топ город (по выручке)

## 4. Таб 2 — Выручка (revenue deep-dive)

**Задача таба.** Ответить: «Где моя выручка растёт, где падает, какой канал продаж главный?»

### 4.1 Above the fold (P0)

**3 KPI с тренд-sparkline:**
- Итого выручка + sparkline выручки по дням (30 последних точек)
- Средний чек + sparkline
- Выручка на одного клиента (AOV / unique customers) — KPI LTV-прокси

### 4.2 Первый fold (P1)

**Stacked area chart: выручка по payment_mode по дням.**
- X: date
- Y: total revenue (stacked)
- Series: PREPAID (зелёный), PAY_WITH_CREDIT (фиолетовый)
- Legend сверху
- Tooltip показывает абсолютные значения + % от whole

**Почему stacked, а не grouped.** Нам важно два инсайта одновременно: (а) общий тренд выручки, (б) как распределяется между наличкой и кредитом. Stacked решает оба.

### 4.3 Второй fold (P1)

**Два donut chart рядом** (сетка 2 колонки):

| Donut 1: По способу оплаты | Donut 2: По способу доставки |
|---|---|
| PREPAID vs PAY_WITH_CREDIT | PICKUP vs LOCAL vs REGIONAL_TODOOR |
| Зелёный / Фиолетовый | Серый / Синий / Оранжевый |

**Почему donut OK здесь.** 2–3 категории с явным whole (100% выручки). Центр donut показывает общую сумму. Pie chart всё равно не используем — donut лучше читается.

### 4.4 Ниже фолда (P2)

**Bar chart: средний чек по периодам.**
- Показывает стоял / рос / падал средний чек
- Horizontal reference line = median всего периода

## 5. Таб 3 — Заказы (operational health)

**Задача таба.** Ответить: «Где я теряю заказы и в какое время пика спроса?»

### 5.1 Above the fold (P0)

**Grid 2 × 3 pills** со статусами заказов:

| COMPLETED | CANCELLED | RETURNED |
|---|---|---|
| 1 342 · 87% | 133 · **8.6%** | 59 · 3.9% |
| зелёный фон | красный если ≥3% | оранжевый если ≥2% |

| ACCEPTED_BY_MERCHANT | NEW | KASPI_DELIVERY |
|---|---|---|
| 7 · 0.5% | 0 · 0% | 99.8% |

**Цветовой код.** COMPLETED — единственный зелёный. Все «в процессе» — серый. CANCELLED / RETURNED — красный/оранжевый только при превышении порога Kaspi (3% и 2% соответственно).

### 5.2 Первый fold (P0)

**Line chart: cancellation rate + return rate по времени (dual line).**
- Y: проценты (0–15%)
- 2 линии: отмены (красная) и возвраты (оранжевая)
- **Reference lines**: горизонтальные пунктиры на уровне 3% (Kaspi threshold отмен) и 2% (возвратов)
- Когда линия пересекает reference — фон области выше подсвечивается тонким красным

**Почему reference lines критично.** Kaspi блокирует магазин при превышении. Клиент должен видеть «я близок к проблеме» не вычисляя.

### 5.3 Второй fold (P1) — парные графики

**Сетка 2 колонки:**

**Bar: заказы по дням недели.**
- X: Пн–Вс
- Y: кол-во заказов
- Max bar подсвечивается зелёным
- Min — серым

**Bar: заказы по часам.**
- X: 0:00 → 23:00
- Y: кол-во заказов
- Shade: 9:00–21:00 (активные часы) — без подсветки; часы вне — muted

**Почему bar, а не line для time-of-day.** Дискретные часы — bar. Непрерывное время (дни) — line. Нарушать это — confuse user.

### 5.4 Ниже фолда (P2)

**Донат заказов по статусам.** Дубликат pill-grid в визуальном виде — для презентаций / скриншотов клиенту.

## 6. Таб 4 — Доставка (geo + cost)

**Задача таба.** Ответить: «Где мои покупатели живут и во сколько мне обходится доставка?»

### 6.1 Above the fold (P0)

**4 KPI:**
- Общая стоимость доставки для продавца
- Средняя стоимость доставки / заказ
- % экспресс-доставок
- % Kaspi Delivery (vs Моя доставка)

### 6.2 Первый fold (P0) — **KZ Heatmap**

**SVG-карта Казахстана, 17 областей + города республиканского значения.**
- Цветовая шкала: белый → розовый → красный (white → #fee2e2 → #b91c1c)
- Metric toggle (segmented control): `Выручка / Заказы / Средний чек`
- Hover на область → tooltip: название, выручка, % от всего рынка продавца, кол-во заказов
- Активная область подсвечивается тонкой обводкой
- **Легенда шкалы** внизу карты: min value → max value (числа реальные, не абстракт)

**Почему heatmap, а не bar list.** География воспринимается пространственно. Карта мгновенно показывает «моя выручка сконцентрирована в Алматы и Астане, регионы пустые». Bar list тратит 10 секунд на чтение.

### 6.3 Второй fold (P1)

**Horizontal bar chart: топ-20 городов.**
- Y: название города (sorted by revenue desc)
- X: выручка (bar) + отдельная колонка с заказами
- Каждый bar — два сегмента visualized на одной линии: выручка (широкий) + заказы (секондарная шкала)
- Альтернатива проще: одна bar-чарт выручка + справа колонка с текстовыми числами заказов

**Почему horizontal, а не vertical.** Длинные русские названия городов не влезают под vertical bars, приходится крутить labels на 45° — нечитаемо.

### 6.4 Ниже фолда (P2)

**Сетка 2 колонки:**

**Stacked bar: стоимость доставки по периодам (месяцы).**
- X: месяц
- Y: стоимость
- Stack: стоимость для продавца (серый) + стоимость для покупателя (голубой)

**Donut: экспресс vs обычная доставка.**
- 2 категории, нормально для donut

## 7. Таб 5 — Клиенты (retention + LTV)

**Задача таба.** Ответить: «Возвращаются ли мои покупатели и кто приносит больше всего денег?»

### 7.1 Above the fold (P0)

**4 KPI:**
- Уникальных клиентов
- Среднее заказов на клиента (retention proxy)
- LTV средний (выручка / уникальных клиентов)
- % повторных покупателей (клиенты с ≥2 заказами)

### 7.2 Первый fold (P0)

**Двойная гоистограмма: новые vs возвращающиеся клиенты по месяцам.**
- X: месяц
- Y: кол-во клиентов
- Series: новые (голубой) + возвращающиеся (зелёный)
- Side-by-side grouped bar (не stacked — нужно видеть каждую категорию)

**Почему grouped, а не stacked.** Важно видеть **отдельно** динамику привлечения новых (marketing health) и retention (product-market fit). Stacked скрывает падение новых если retention растёт.

### 7.3 Второй fold (P1)

**Таблица «Топ-20 клиентов»** — 4 колонки:

| Имя | Заказов | Выручка | Средний чек |
|---|---:|---:|---:|
| Ольга Х. | 12 | 348 000 ₸ | 29 000 ₸ |

Sorted by revenue desc. **Phone не показываем** (Kaspi обфусцирует всё равно).

### 7.4 Ниже фолда (P2)

**Bar: распределение клиентов по кол-ву заказов.**
- X: 1 заказ / 2 заказа / 3–5 / 6–10 / 11+
- Y: кол-во клиентов
- Видна «one-timer proportion» — часто 80% клиентов купили только раз.

**Gauge / progress bar**: retention rate (% клиентов с ≥2 заказами от всех). Goal-like visualization — стрелка показывает где магазин vs бенчмарк 25% (средний для e-commerce).

- Border: `#1e1e30`

## 12. Типографика

- **Числа**: tabular-nums везде (одинаковая ширина цифр)
- **KPI-значения**: 24–28px, font-extrabold
- **KPI-labels**: 10–11px, uppercase, letter-spacing 0.08em, #999
- **Chart titles**: 13–14px, font-semibold, #e8e8f0
- **Chart axes**: 10–11px, #999
- **Body**: 13px, #ccc
- **Tooltips**: 12px, контрастный фон

Шрифт: системный sans-serif (`ui-sans-serif, system-ui, -apple-system, Segoe UI, Roboto`). Никаких Google Fonts для MVP — latency не оправдан.

## Приложение А — почему не пай-чарты

Любой pie chart в этом проекте — запрещён, **кроме donut с ≤4 сегментами**. Причина: человек плохо сравнивает углы (3× хуже чем длины баров). Для 5+ категорий используем horizontal bar chart сортированный по убыванию.

## Приложение Б — почему не 3D

3D-графики никогда. Искажают восприятие (передний сегмент всегда кажется больше). Каноны Few, Tufte категоричны.

## Приложение В — почему не multiple Y-axes везде

Dual-axis только когда две метрики **связаны смыслом** (выручка и заказы — да, корреляция ожидаема). Две несвязанные метрики на одной оси — визуальный трюк, запрещён.
